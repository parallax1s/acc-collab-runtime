# ACC Claude Plugin Portable Bootstrap Design

## Goal

Turn the current thin Claude packaging in `acc-collab-runtime` into a portable, inspectable integration that installs once globally and bootstraps repo-local ACC behavior on demand.

## Problem

The public repo currently installs a Claude skill, registers a local plugin, and exposes markdown slash commands, but it does not package the stronger Claude behavior that exists in `Werkstatt`.

Missing behavior includes:

- repo-local `.agent-comms` initialization
- repo-local Claude hook installation
- conservative merge into repo-local Claude settings
- portable hook generation that avoids project-specific absolute paths

As a result, the Claude plugin is discoverable but not operationally complete.

## Non-Goals

- multi-Claude-agent support in the first version
- a fully global hook that auto-discovers ACC state across repos
- hidden runtime behavior outside repo-visible files
- overwriting arbitrary user-managed Claude hook configuration

## Recommended Approach

Use a hybrid packaging model:

- global install once into `~/.claude`
- repo-local bootstrap for each project that wants ACC

This keeps the skill and plugin globally available while making the actual coordination wiring explicit, debuggable, and repo-scoped.

## Architecture

### Global Install

The global Claude installer remains responsible for:

- installing the ACC skill into `~/.claude/skills/acc-collab-runtime`
- registering the local plugin in Claude plugin metadata
- exposing ACC slash commands and bootstrap entrypoints

The global installer must not write repo state.

### Repo Bootstrap

`acc install --type claude --agent-id <id>` becomes the portable bootstrap path for Claude repos.

Bootstrap is responsible for:

- creating `.agent-comms/` if missing
- creating any missing comms subdirectories
- writing `.claude/hooks/acc-pre-tool-use.js` in the current repo
- conservatively merging ACC hook configuration into repo-local `.claude/settings.json`
- recording the selected single-agent identity in generated hook output

### Runtime Hook

The generated Claude hook handles runtime enforcement only:

- deny writes when another active lease covers the target file
- inject inbox context into Bash and write-capable tool calls
- use repo-local `.agent-comms` as the source of truth

The hook should not perform installation, migration, or plugin discovery logic.

## File Responsibilities

### Existing Files To Extend

- `scripts/install-claude-skill.sh`
  - keep as the one-time global installer
  - clarify that repo setup happens through ACC bootstrap

- `acc/src/commands/install.js`
  - add the repo bootstrap flow for Claude
  - create `.agent-comms`
  - install the managed hook file
  - merge repo-local Claude settings

- `README.md`
  - explain the split between global install and repo bootstrap
  - document what files bootstrap writes into a repo

### New Files To Add

- `acc/src/hooks/claude-pre-tool-use.js`
  - portable hook template or generator source
  - must avoid hard-coded repo-specific paths from `Werkstatt`

- `acc/src/core/claude.js`
  - helper functions for bootstrap, settings merge, and managed-file detection

- test fixtures for repo-local Claude settings merge scenarios

## Managed File Strategy

ACC should only overwrite files it clearly owns.

### Hook File

Generated `.claude/hooks/acc-pre-tool-use.js` should include a stable managed marker such as:

- `Generated by ACC install — do not edit manually.`

Bootstrap behavior:

- if the hook file is missing, create it
- if the hook file exists and has the ACC managed marker, replace it in place
- if the hook file exists without the ACC marker, stop and report the conflict

### Settings Merge

ACC should preserve unrelated settings and unrelated hook config.

Bootstrap behavior:

- if `.claude/settings.json` is missing, create it with the ACC hook block
- if it exists and is valid JSON, merge conservatively
- if it is malformed JSON, stop without writing
- if the hook layout is ambiguous or would require overwriting unrelated config, stop with an exact explanation

ACC-managed hook entries should be identifiable so reruns are idempotent.

## Idempotence

Running bootstrap multiple times in the same repo should:

- keep `.agent-comms` intact
- recreate only missing comms subdirectories
- refresh ACC-managed hook output safely
- preserve unrelated Claude settings
- avoid duplicate hook registrations

## Single-Agent First Version

The first portable version should optimize for one Claude agent per repo.

Implications:

- bootstrap accepts one `--agent-id`
- generated hook embeds one agent identity
- docs should state that multi-agent Claude support is a later extension

This avoids premature complexity in settings merge and runtime routing.

## Error Handling

Bootstrap must fail loudly and specifically.

### Hard Stop Cases

- malformed `.claude/settings.json`
- unmanaged existing hook at `.claude/hooks/acc-pre-tool-use.js`
- unsupported or ambiguous hook structure in existing settings
- inability to create required repo directories or files

### Error Output Requirements

Errors should report:

- the exact file path involved
- why ACC refused to proceed
- the next manual action when applicable

Bootstrap must not leave partially updated config behind after a hard stop.

## Migration From Current Public Repo

The current Claude plugin install path stays valid, but the README and command behavior should shift:

- global install only makes ACC available to Claude
- repo bootstrap makes ACC active in a repository

This preserves current install expectations while making the operational boundary explicit.

## Testing Strategy

### Installer Coverage

Keep existing user-home installer tests for:

- skill symlink creation
- plugin registration
- settings enablement under `~/.claude`

### Bootstrap Coverage

Add tests for:

- fresh repo bootstrap creates `.agent-comms`, hook, and settings
- rerunning bootstrap is idempotent
- valid existing settings are merged without clobbering unrelated keys
- malformed settings cause refusal without writes
- unmanaged hook file causes refusal
- ACC-managed hook file is updated in place

### Runtime Hook Coverage

Add or adapt tests to verify:

- leased files are denied when owned by another agent
- inbox context is injected when pending messages exist
- repo-relative file canonicalization works without absolute-path assumptions

## Success Criteria

The Claude plugin is considered finished for this slice when:

- a user can install ACC once into Claude
- a user can bootstrap a fresh repo with one command
- the repo gains visible `.agent-comms` and `.claude` ACC wiring
- rerunning bootstrap is safe
- existing user config is preserved unless ACC cannot merge safely
- the implementation no longer depends on `Werkstatt`-specific paths
